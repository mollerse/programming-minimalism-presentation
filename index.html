Programming Minimalism

Sometimes programming is to us as water is to the goldfish. It's easy to take things for granted. Or get stuck in thinking we've solved it.

A little careful consideration and exploration is both fun and neccessary every now and then.

(As well as reminding ourselves «what water is».)

Programming Minimalism

Minimalism at it's core isn't about owning the fewest things. Even if you could do it.

And therefore I'm not going to Marie Kondo your favoriteTM PL or try to convince you that there is some PL out there that is more virtuous for some measure of virtue.

(Code is immaterial, it would be pointless.)

(You'll get that pun later)

If we're going to be taking apart a PL, we need to have some idea of what a PL is made up of; enter syntax and semantics.

Syntax dictates the form of a progam.

Semantics dictates the meaning of a program.

Syntax & Semantics

Syntax and semantics are deeply interconnected. If you want a thing to have a different behavior in a different context you usually reach for syntax to make that happen.

And especially in PLs, where relying purely on contextual information is highly impractical.

Aside: Semiotics

<js>


              class Dog() extends Animal {

Eventhough jølle is a dog in this program, jølle isn't actually a dog.

(«This is not a pipe» and all that.)

Aside: Semiotics

jølle is an abstract representation of a dog. A symbol if you will.

We only interpret jølle as a dog because we have an idea of what a dog is; an animal that barks and is a good boy.

And that's semiotics. Not semantics.

Say hello to
«Concatenative Programming»!

Concatenative programming

Programs are constructed by composing functions.

As opposed to the way most other PLs work, which is by applying functions to values.

(Concatenative vs Applicative.)

Concatenative programming

There are no values, atleast not in the traditional sense, only functions.

What looks like values are actually functions that produce that value.

(Who knew "" could be a function.)

Concatenative programming

Every function has the same type: f State => State

State is usually a stack, because stacks are simple.

Concatenative programming

Because of the global state usually lives on a stack, concatenative programming is often referred to as «Stack Based Programming».

Personally I don't like that. Too much in common with the «Turing Tarpit».

Besides, the stack is the least intresting part.

Concatenative programming

These languages are not a modern phenomenon. They've been around for a while.

Forth (1968) and Joy (2001) are probably the two most well known.

Which also happen to represent two very different approaches.

Forth

Created by Charles «Chuck» H. Moore to program directly on CPUs, as an alternative to Assembly.

Designed to be simple, portable and human-friendly.

Has seen wide and varied use.

Joy

Created by Manfred von Thun because he wanted a simpler Lambda Calculus without variable bindings.

Inspired by Backus' FP and SKI calculus from combinatory logic.

Concatenative programming

The funny thing is, Forth and Joy both ended up in the same place, despite following vastly different paths.

That says something about the power of simple semantics driving a simple syntax.

Concatenative Geometry

We have a PL that's supposedly excellent at composing programs.

What better than to implement some of the stuff from Peter Henderson's seminal 1982 paper «Functional Geometry»

(Actually, we're going to use the 2002 revised version.)

Concatenative Geometry

The basis of the paper is to construct a tessellation.

Which is math-speak for covering a plane with repeating images that completely cover the entire plane.

Concatenative Geometry

We are going to be doing this using HTML Canvas API and a concatenative language implemented in JS.

(And yes, I've committed the sin of implementing my own language. More on that later.)

Concatenative Geometry

The paper outlines three basic operations: flip, turn and toss.

In addition the paper defines over, which groups images that belong to the same box. We won't need that, for reasons that hopefully becomes apparent.

Concatenative Geometry

Next up the paper defines a couple of tesselation tiles: ttile and utile

ttile is helpfully defined as:


                over(draw, over(flip(toss(draw)), turn(turn(turn(draw)))))

and utile:


                over(over(flip(toss(draw)), turn(flip(toss(draw)))), over(turn(turn(flip(toss(draw)))), turn(turn(turn(flip(toss(draw))))))

Concatenative Geometry

We now have all the parts we need to make our tessellation.

We'll divide the plane into a 3x3 grid and fill each cell with a tile such that it completely covers the whole canvas.

Concatenative Programming

Quotation. A sub-program omitted from execution.

We can modify quotations using concatenative operations like cons, append.

Gives us the ability to do all sort of control flow, higher order functions, and recursive shennanigans.

In praise of simplicity

What suprises me the most is how little we actually lose.

Sure, we have to do some deeply dissatisfying stack fiddling.

And operand odering for operators like / and % makes my brain hurt a little.

In praise of simplicity

The meta-programming you can do with quotations reminds me of LISP.

Wanna partially apply an argument to a bunch of functions? cons & map

In praise of simplicity

Explorative programing, like what you would do when working creatively, feels really good.

The lack of syntax getting in your way is *chef's kiss*

Also reordering the functions in a pipeline or slipping a function call into the middle of a pipeline is really neat.

In praise of simplicity

The runtime for a PL like this is suprisingly easy to make.

There's a reason you can find PLs with similar semantics in the JVM (JVM Byte Code) and in printers (PostScript).

In praise of simplicity

Given how easy it is to create a runtime for a PL like this, making DSLs and embedding them in your program as an alternative to a GUI seems like a good idea.

Especially for things that already rely on shared global state, like HTML Canvas or WebAudioAPI.
